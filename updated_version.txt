version 2.2
1. 静态避障路径规划
   - 确定凸空间（PathBoudary）：结合obs边界和道路边界进行初步拓展，再由 ADC_bound 和 lat_buffer 对凸空间进行收缩，
     收缩后判断可通行性，如果边界收缩后不可通行，则会在阻塞区前10m停车，防止在较远处感知边缘box形状畸变导致的误判

   - 通过obs的绕行决策提取obs指定角点取代AABB法，obs边界贴合自身几何形状

   - FrenetQPPlanner约束条件：分段 jerk QP，约束含边界、曲率、jerk、连续性、初始状态、ego 四角点。
     其中ego角点约束用的是收缩前的PathBoudary，防止约束过于严苛

   - carla_vehicle_control逻辑优化（原先逻辑为跟踪 path_smoother 平滑后的参考线）：
     优先跟踪 QP 局部路径，参考线仅作为坐标转换的参考系和最次解；
     采用双索引（ref_nearest_idx / traj_nearest_idx）将局部路径与参考线解耦

   - _check_path_reusable：根据obs更新情况和ego与上一周期的位置偏差判断是否复用QP局部路径，减小算力开销。
     需要与复用的QP轨迹对齐，更新时由上一次复用的QP轨迹获取(l_init, dl_init, ddl_init)三个初始值，供新QP的起点约束使用

   - 仅由复用QP插值得到的(l_init, dl_init, ddl_init)估计值可能不准确（并非复用QP求得的原始点），
     所以可能复用QP可以得到的解，在新的QP求解中就因初始点约束太过严格（最初为等式约束）且速度、参考线曲率发生一定变化，
     导致其与其它约束（比如角点约束、曲率约束）产生冲突，有概率会求解失败

     主要原因是贴近障碍物时求解空间减小，因为纵向步长确定的情况下，横向步长会受曲率约束影响
     （失败大多存在于经过obs时，由osqp_status=7或2转为3）

     所以当前起始点约束采用的是软约束，允许极小范围的偏差调整，能极大降低求解失败概率（数十次测试下，由原本的10%到0%）。

   - 修正 PathBoundary 逻辑：@map_load/map_load/vehicle_perception.py:912-915，修正错画边界的逻辑

   - vehicle_plotter：s-l 图绘制 PathBoundary 与 QP 路径

version 2.2.1
1. 修改避障局部规划中QP的cost函数。原本的前两项（w_ref, w_l所在项）都是为了尽可能加强规划的轨迹和参考线的贴合程度。
   但是当obs侵入参考线时，求解器为了贴合参考线，导致规划的轨迹都是贴合obs边界的，这其实并不利于安全。
   此外，这两项功能相同设计上存在冗余，所以选择删去后者，更改前者，使其更为贴近凸空间的中心线。
   改进后可以较大程度地提升避障轨迹的安全性，理论上也可以提升求解成功率

2. 当前规划的参考线其实并非沿着道路中心线走的，而是在道路的1/4处。因为当前道路是双车道，两条车道沿道路1/2处分开，
   参考线是处于下方道路的中心线，而上方的道路可能存在对向来车，为了对应上一点的更改（贴近凸空间的中心线），
   并不能时刻都把凸空间上界定义为整条道路的上界，而是仅在需要借道时拓展至该位置，以保证我们ego不侵入上方的对向车道。

3. 修改carla-ros-bridge逻辑，补充显式RTF设置，便于调整 sim-time 和 wall-time 之间的比值
   在两次tick中加入节流逻辑来实现RTF设置
   （需要注意carla为service端，carla-ros-bridge和其他ROS节点属于一个client端。
   client端设置同步模式时需要靠 world.tick() 推进仿真时间，且需确保只能有一个client可以主动tick，
   其余client需要 world.wait_for_tick() 来保持同步（如scenario_set））

   由gazebo的变量定义可推导得：
   sim_time_elapse / wall_time_elapse = max_step_size * real_time_update_rate = real_time_factor(RTF)
   详见 REAL_TIME_FACTOR_CHANGES.md

   ROS相关节点也需要进行设置上的更改，确保订阅的是carla-ros-bridge发布的 '/clock' 话题，
   self.get_clock().now() 才不会获取到默认订阅的ROS系统时间(wall-time)，保证时钟源统一：
   super().__init__(
      'ROS_NODE_NAME',
      parameter_overrides=[Parameter('use_sim_time', value=True)],
   )

   需要承认的是，先前的版本一直没有发现这个问题的存在。因为 object.json 文件前几次设置了6个摄像头且分辨率和帧率较高，
   且经过订阅 '/clock' 话题发现启动 Terminal 3(spawn objects) 后，受CPU算力影响，
   其频率(real_time_update_rate)正好在20Hz左右，
   又因为carla-ros-bridge默认设置的 fixed_delta_seconds(max_step_size)为0.05，
   所以主观上难以察觉 “sim-time 和 wall-time不同步” 的问题，当然也是受限于先验知识的积累
   希望对今后的学习有所警示，提升对潜在问题的洞察能力和基础知识的重视

6. 获取碰撞区域，投影到ST图（速度规划前置工作）
   - ST图投影参考线为QP算出的局部路径，以ego位置为起点（需对复用的路径进行裁剪）

   - Apollo的方案是在 Cartesian 坐标系靠两层循环分别迭代obs和ego的轨迹，判断两者box是否有碰撞
     外层迭代obs轨迹的trajectory_step，内层迭代ego的s，得到粗解后二分得到首次碰撞和最后一次碰撞的s

     我选择只遍历obs的Trajectory，把该Trajectory的每一个dt在Cartesian坐标系下先转换获得其box，
     然后投影到新参考线的frenet坐标系上；ego的轨迹投影到新参考线frenet坐标系上就是两条平行于s轴的线段，
     两者的l分别为正负adc_buffer，长度定义为上述Trajectory的s长度加上ego的半个车长（考虑ego的box）
     而这两条平行的线段和adc_buffer正好组成一个新矩形的长和宽

     问题就转换为用这个新矩形去裁剪obs投影形成的多边形，求裁剪后多边形的min_s和max_s（肯定处于端点处）
     对于该典型问题，我采用的是Sutherland–Hodgman算法（计算机图形学）

     其实最开始是打算用扫描线算法求解obs多边形内部平行于S轴的最长线段并将其视为结果，
     分析后发现其仅使用于部分场景（如obs垂直于参考线切入或只有一个角点位于新矩形内），结果极具局限性。
     我们常希望能尽快用已知的知识体系去转化和解决问题，但遇到素为谋面的问题时，这往往是最大的局限所在

   - 定义新消息类型，见map_load/msg。将 Path 和 SpeedProfile 对齐得到 Trajectory， 
     Trajectory 替代原本的 Path 发布至控制器。便于后续路径和速度的解偶规划

   - 新参考线路径可能存在较大曲率；obs四个角点各自投到参考线得到相应坐标后，
     在 s–l 平面用直线连接四点可能不等价于 box 在真实空间占用的区域，边可能顺序错乱或得到凹多边形。
     采用Apollo “边中点 + 补点” 的流程（ReferenceLine::GetSLBoundary）增强其泛化能力
